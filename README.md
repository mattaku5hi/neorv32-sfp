# neorv32-sfp

Прошивка для программного RISC-V ядра **NEORV32**, интегрированного в FPGA
Gowin GW2A. Ветка реализует *TWI/GPIO-шелл* — интерактивную консоль поверх
UART, через которую пользователь набирает транзакции I²C и команды GPIO, а
прошивка их парсит, исполняет на шине и возвращает результат на терминал.

Код написан на чистом C без `<stdlib.h>` и без плавающей точки:
собственные `printf` / `snprintf` / построчный ввод из `<inc/neo_io.h>` и
полностью host-тестируемый парсер/исполнитель.

---

## 1. Состав

| Модуль | Где лежит | Назначение |
|--------|-----------|------------|
| `neo_io`     | `inc/neo_io.h`, `src/neo_io.c`       | Минимальные `printf` / `snprintf` / `read_line` с подключаемым backend'ом (UART на таргете, строковый буфер в тестах). Никакого `<stdlib.h>`, никакой float. |
| `neo_uart`   | `inc/neo_uart.h`, `src/neo_uart.c`   | NEORV32 UART0 как backend для `neo_io` + построчный ввод до 1024 байт. |
| `neo_parser` | `inc/neo_parser.h`, `src/neo_parser.c` | Парсер и валидатор транзакций (грамматика ниже). Не зависит от NEORV32 — собирается на хосте. |
| `neo_exec`   | `inc/neo_exec.h`, `src/neo_exec.c`   | Исполнитель: TWI + GPIO HAL (vtable), чтобы можно было подменить на мок в тестах. |
| `neo_app`    | `inc/neo_app.h`, `src/neo_app.c`     | Верхний REPL: инициализация, цикл «prompt → read_line → parse → exec». |
| `main`       | `src/main.c`                         | Точка входа прошивки, баннер, запуск REPL. |
| `test/`      | см. `test/README.md`                 | Хост-сборка юнит-тестов (CMake + CTest). |

## 2. Грамматика транзакции

Транзакция — одна строка, оконченная `Enter`. Не более **1024 байт**.

### 2.1 Формат байта в тексте команды

| Префикс | Формат | Пример | Требования к длине |
|---------|--------|--------|--------------------|
| `h` | шестнадцатеричный | `hA0`, `hDE` | кратно 2 символам, только `[0-9A-F]` (верхний регистр!) |
| `b` | двоичный          | `b10010101` | кратно 8 бит, `_` допускается как визуальный разделитель |
| `a` | ASCII (латиница + цифры) | `aCentin_` | обязательно завершается `_` |

> **Почему hex только в верхнем регистре?** Чтобы отличать маркер формата
> `b` / `a` / `h` (строчные) от hex-цифр.

`_` между группами разных форматов всегда допустим. Любые другие символы —
ошибка.

### 2.2 TWI-команды

```
hA0|hA2 <iaddr> <payload>     — запись, 0..1022 байт данных
hA1|hA3 <iaddr> h<count><h|b|a>_   — чтение до h3FE (1022) байт
```

- `<iaddr>` — один байт внутреннего адреса устройства в любом формате
  (`h00`, `b00000000`, ...).
- При записи `payload` может быть последовательностью групп разного
  формата: `hA0_b10010101h09E7b0011_010_11111_1_011_aCentin_h4E`.
- При чтении `<count>` — 1..3 hex-цифры; `h0` (нуль-байт) допускает
  пропуск формата, иначе формат обязателен. Команда чтения обязана
  заканчиваться `_`.
- Байт `A0`/`A1`/`A2`/`A3` **внутри** команды обязан быть в двоичном
  или ASCII виде, иначе он будет воспринят как начало следующей команды.
- Несколько TWI-команд в одной транзакции — цепочка с repeated-START
  (STOP только в самом конце).

### 2.3 Специальные команды

| Команда | Действие |
|---------|----------|
| `out_xxxxxxxx` | Записать ровно 8 бит в GPIO-порт |
| `inp`          | Прочитать байт из GPIO, вывести как `gpio => xxxxxxxx` |
| `help` / `?`   | Краткая справка по командам и форматам |

### 2.4 Вывод на терминал

- **Запись**: каждый переданный байт эхом в hex + `+` (ACK=0) или `-` (NACK=1).
- **Чтение**: `ctrl` и `iaddr` — с ACK-знаком, как у записи; далее данные —
  в запрошенном формате **без** ACK-знака. Если на данном байте был NACK,
  вместо байта печатается `-`.
- Каждая команда — с новой строки. После транзакции снова `> ` — приглашение.

Канонический пример из ТЗ:

```
hA0_b10010101h09E7b0011_010_11111_1_011_aCentin_h4E
```

Одна команда записи: `ctrl=A0`, `iaddr=10010101`, payload
`09 E7 35 FB 43 65 6E 74 69 6E 4E` (hex + bin + bin + ascii + hex).

## 3. Быстрый старт

### 3.1 Зависимости

```bash
scripts/build.sh --install
```

Устанавливает через `apt` хостовые пакеты: `build-essential`, `cmake`,
`ninja-build`, `pkg-config`, `git`. Опция `--install` **эксклюзивная**: она
требует sudo, поэтому скрипт гарантированно не создаст ни одного билд-артефакта
от root'а — чтобы запустить сборку, перевызовите скрипт уже без `--install`.

RISC-V toolchain нужен отдельно (для прошивки). По умолчанию ожидается
в `/opt/riscv/bin/riscv32-unknown-elf-`; путь переопределяется опцией
`--toolchain`.

### 3.2 Клонирование

```bash
git clone <repo-url> neorv32-sfp
cd neorv32-sfp
git submodule update --init --recursive
```

### 3.3 Сборка прошивки

```bash
scripts/build.sh                              # release
scripts/build.sh --debug                      # debug
scripts/build.sh --toolchain ~/riscv/bin/riscv32-unknown-elf-
```

Артефакты: `build/release/app/` (или `build/debug/app/`) — `.elf`, `.bin`,
`.hex`, `.asm`, `.vhd` и производные образы для загрузки в FPGA.

### 3.4 Тесты

```bash
scripts/build.sh --tests        # прошивка + хост-тесты
scripts/build.sh --tests-only   # только хост-тесты
```

Тесты — отдельный нативный CMake-подпроект в `test/`, включается опцией
`-DBUILD_TESTS=ON` корневого CMake. В этом режиме корневой `CMakeLists.txt`
пропускает кросс-сборку прошивки и конфигурирует только тесты. Запуск —
через CTest (`ctest --test-dir build/tests --output-on-failure`).

Подробная инвентаризация кейсов — в `test/README.md`.

### 3.5 Полный справочник build-скрипта

```bash
scripts/build.sh --help
```

Ключевые опции:

| Опция | Что делает |
|-------|------------|
| `--release` / `--debug` | Конфигурация прошивки (по умолчанию release) |
| `--tests`               | Дополнительно собрать и прогнать хост-тесты |
| `--tests-only`          | Собрать и прогнать только хост-тесты |
| `--toolchain <pfx>`     | Переопределить `TOOLCHAIN_PREFIX` |
| `--install`             | Установить хост-зависимости (эксклюзивно, exit) |
| `--clean`               | Почистить `build/` перед конфигурированием |
| `--no-clean`            | Не чистить `build/` |
| `--jobs N` / `-j N`     | Число параллельных задач (по умолчанию `nproc`) |
| `--build-dir <dir>`     | Переопределить корневой каталог сборки |
| `--verbose`             | Детальный вывод сборки |

## 5. Ограничения и допущения

- Hex-цифры только в верхнем регистре (`[0-9A-F]`) — см. §2.1.
- Максимум команд в одной транзакции: `NEO_PARSER_MAX_CMDS = 160`.
- Максимум payload одной записи: 1022 байта (по спеке).
- `printf`-реализация не поддерживает плавающую точку — аппаратного FPU
  нет и расширения компилятора тоже.
- Тесты зависят только от хостового gcc; никакой сетевой активности,
  никаких внешних фреймворков.
